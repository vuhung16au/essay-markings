# Documentation Policy

## Goals

- Keep docs aligned with the codebase.
- Maintain a clear separation between official docs and scratch notes.

## Rules

- Project-facing guidance belongs in `docs/`.
- Temporary planning notes should not be treated as canonical references.
- Use relative links in Markdown.
- Update docs when rubric, scoring behavior, exports, or configuration changes.
- Avoid leaving outdated examples after implementation changes.

## Preferred docs to keep current

- API behavior
- scoring behavior
- calibration workflow
- configuration
- testing steps
- export behavior
